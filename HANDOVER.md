# Storekeeper — складской учёт и продажи для типографии/наружной рекламы

Полное состояние проекта и всё, что реализовано. Этот файл — точка входа для
продолжения работы в новом чате.

> Бренд: **Storekeeper** (ранее «Cloude»). Стек: **Django 4.2 + DRF + JWT**

## 3. Склад: рулоны, листы, единицы

Материалы приходят в разных формах, всё считается в **кв.м** (для площадных):
- **Лист** (акрил 1.20×2.40): площадь = ширина × высота × кол-во листов.
- **Рулон** (форекс, самоклейка): ширина фикс, длина РАЗНАЯ → площадь = ширина × длина.
- **По количеству** (клей, краска, крепёж): кг/л/шт, без площади.

Модель `Material`: `unit` (SQM/METER/PIECE/KG/LITER), `is_roll_material` (площадной).
Модель `Roll` (партия/лот): `form` (ROLL/SHEET), размеры, `initial/remaining_area`
(кв.м), `purchase_cost`, `markup_percent` → property `price_per_sqm` (= cost×(1+%)/area),
`cost_per_sqm`. **Наценка задаётся при каждом поступлении.**

`warehouse/rolls.py`:
- `receive_lot(...)` — создаёт партию, считает площадь, обновляет остаток материала
  и его розничную цену (= по последнему рулону).
- `consume_area(...)` — списывает кв.м **FIFO** по партиям (старые первыми).
- `restore_area(...)` — возврат площади (рефанд).

UI приёма — единый экран **«Поступление»** (страница Supply складовщика) с
сегментом **Рулон / Лист / По количеству** + живое превью (площадь, цена/кв.м, «станет»).
Вкладки: Поступление, Инвентаризация, Списание. Списание — причины
порча/брак/утеря/срок (тип лога `WRITE_OFF`, учитывается в потерях дашборда).

Эндпоинты: `/api/warehouse/materials/` (+ actions `receive-roll`, `supply`, `adjust`,
`write-off`, `update-price`), `/api/warehouse/rolls/`, `/api/warehouse/inventory-logs/`.

## 4. Услуги: резка, установка, виды букв

`PrintingService.kind`:
- **CUTTING (резка)** — по кв.м. Ставка зависит от вида букв: `rate_flat` (плоские) /
  `rate_volumetric` (объёмные). Цена клиенту = **ставка × кв.м + цена материала × кв.м**
  (работа + материал). Материал (лист/рулон) выбирается при продаже, списывается по площади.
- **INSTALL_EXTERIOR (наружная установка)** — `rate_per_piece` × кол-во букв.
  Крепёж НЕ списывается (только работа).
- **INSTALL_INTERIOR (внутренняя установка)** — монтаж на подложку/самоклейку,
  как резка: `rate_flat` × кв.м + материал × кв.м, материал списывается по площади.
- **OTHER / INSTALLATION (legacy)** — фикс цена.

`ServiceRecipe` (техкарта доп. материалов): `material`, `consumption_per_unit`,
`applies_to` (ALL/FLAT/VOLUMETRIC), `consumption_mode` (PER_SQM/FIXED).
Пример: объёмные буквы дополнительно списывают **клей** (0.05 л/кв.м, applies_to=VOLUMETRIC).

Свойства услуги: `uses_area`, `uses_letter_type`, `uses_pieces` — фронт по ним строит UI.

Расчёт и списание — `sales/sale_service.py::_build_item` и `_deduct_stock_for_item`.

## 5. Продажи (POS), возвраты, дозаказ

**Оформление продажи** (`/app/checkout`) — POS-стиль:
- Слева сетка товаров/услуг (тап → в чек). Материал — степпер кол-ва; резка/внутренняя
  установка — конфигуратор (материал + ширина×длина [+ вид букв для резки]); наружная —
  степпер по буквам; фикс-услуга — кол-во.
- Справа живой чек + блок клиента (живой поиск по телефону, «Кто привёл» для нового
  клиента) + оплата (Наличные / Онлайн) + крупный «Оформить».

**Оплата:** наличные → сразу `PAID` + списание. Онлайн → `PENDING` + платёжная ссылка +
**QR-код** (`payment_qr`, data-URI); списание после подтверждения (вебхук).

**Статусы:** `payment_status` (PENDING/PAID/REFUNDED/PARTIALLY_REFUNDED),
`fulfillment_status` (PROCESSING → READY → ISSUED, экшены `mark-ready`/`mark-issued`,
шлёт клиенту уведомление).

**Возврат** (`refund`) — возвращает материалы на склад (только если был PAID), статусы
обновляются. Списание/возврат рулонных материалов идёт через FIFO/restore.

**Дозаказ** (`add-items`) — добавить позиции в открытый чек позже (напр. установку):
пересчёт итога, списание новых позиций (если PAID). Закрытый/возвращённый чек → 400.
UI: кнопка «+ Дозаказать» в карточке чека (страница «Чеки»).

> ⚠️ Важный фикс: `Receipt.recalculate_total` использует `.filter(is_returned=False)`
> (не `.all()`), т.к. вьюха грузит чек с `prefetch_related` и кэш устаревал. Ответы
> мутирующих экшенов перезагружают чек через `_fresh_response`.

Эндпоинты: `/api/sales/receipts/` (+ `checkout`, `refund`, `add-items`, `mark-ready`,
`mark-issued`).

## 6. Клиенты (CRM) + рефералы

`Client`: тип PHYSICAL/OSOO, ФИО/компания, телефон (unique), telegram_chat_id,
`referred_by` (FK self, related_name=`referrals`).

- **Рефералы «кто кого привёл»:** в карточке клиента — «Кто привёл» + «Привёл клиентов»
  (список с LTV и суммой). **Реферал фиксируется один раз** — после установки изменить/
  сбросить нельзя (валидация → 400). UI показывает 🔒 если задан.
- Доступ: страница «Клиенты» есть и у **админа**, и у **складовщика** (он чаще работает
  с клиентами). «Кто привёл» можно указать прямо при оформлении продажи нового клиента.
- Статистика: заказы, LTV (gross − refunded), статус Telegram.

## 7. Интеграции (код реальный, нужны токены)

- **Telegram:** алерты персоналу о низком остатке (`notify_low_stock`), клиенту —
  электронный чек, статус заказа, возврат, дозаказ. Бот клиентов: верификация по
  контакту (привязка по телефону), меню «Мои чеки». Без токенов — тихий no-op.
- **Платёжный шлюз:** `PAYMENT_GATEWAY=mock` (dev) или `freedompay` (реальный
  FreedomPay/PayBox — подпись + XML-ack вебхука). Ключи в `.env`.

Всё конфигурируется через `.env` (см. `.env.example`).
